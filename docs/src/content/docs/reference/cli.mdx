kerf follows git's split between **porcelain** (human-facing, may evolve) and **plumbing** (scriptable, stable contract).

## Global flags

These are accepted on every subcommand.

| Flag | Purpose |
|---|---|
| `-v`, `--verbose` | Increase log verbosity. Repeat (`-vv`) for more detail. |
| `--identity-file <PATH>` | Path to an age identity file used to unwrap the DEK on decrypt. Env: `KERF_AGE_KEY_FILE` / `SOPS_AGE_KEY_FILE`. |

Every command that decrypts (`decrypt`, `verify`, `view`, `diff`, `edit`, `exec`, `set`, `unset`, `rotate`, `keys add`) reads the same identity; commands that don't decrypt ignore it.

## Command tree

### Porcelain

```
kerf init    --recipient <type:value>... [-o PATH] [--path-regex REGEX] [--encrypted-regex REGEX] [--mac-all]
kerf keygen  -o <PATH>
kerf encrypt <file> [--output PATH | --in-place] [--age KEY] [--kms ARN] [--gcp-kms ID] [--azure-kv URL] [--encrypted-regex REGEX] [--format FORMAT]
kerf decrypt <file> [--output PATH] [--format FORMAT]
kerf edit    <file> [--format FORMAT]
kerf view    <file> [--path DOTTED.PATH] [--format FORMAT]
kerf diff    <old> <new> [--show-values] [--format FORMAT]
kerf exec    <file> [--format FORMAT] -- <command> [args...]
```

### Recipient management

Recipient changes re-wrap the DEK only; body ciphertexts and the MAC stay byte-identical.

```
kerf keys add    <file> [--age KEY] [--kms ARN] [--gcp-kms ID] [--azure-kv URL]
kerf keys remove <file> [--age KEY] [--kms ARN] [--gcp-kms ID] [--azure-kv URL]
kerf keys list   <file>
```

### Rotation

```
kerf rotate <file> [--reason "<msg>"] [--format FORMAT]
```

### Operational

```
kerf verify <file> [--format FORMAT]
kerf set    <file> <path>          # new value is read from stdin, never argv
kerf unset  <file> <path>
```

### Plumbing (stable)

```
kerf recipients     <file>
kerf metadata       <file>
kerf mac --verify   <file>
kerf path-encrypted <file> <path>   # exit 0 if encrypted, 1 if not — quiet
```

## Usage examples

### Set up a config and a key

```bash
# Generate an age keypair (secret written 0600; recipient printed to stdout)
kerf keygen -o ~/.config/kerf/age.key

# Record a creation rule in .kerf.yaml. Recipients are <type>:<value>:
#   age:age1…  aws-kms:arn:aws:kms:…  gcp-kms:projects/…  azure-kv:https://…
kerf init --recipient age:age1abcdef... -o .kerf.yaml
```

### Encrypt and re-encrypt

```bash
# First encrypt
kerf encrypt secrets.yaml --age age1abcdef... --output secrets.kerf.yaml

# Re-encrypt in place: unchanged values keep their exact ciphertext (the kerf rule)
kerf encrypt secrets.kerf.yaml --in-place
```

### Change one value (one-line diff)

```bash
# Value comes from stdin, so it never lands in your shell history or argv
printf 'new-password' | kerf set secrets.kerf.yaml db.password

kerf unset secrets.kerf.yaml api.legacy_token
```

### Read and inspect

```bash
kerf view secrets.kerf.yaml --path db.password   # one value to stdout
kerf decrypt secrets.kerf.yaml                    # whole file to stdout
kerf diff old.kerf.yaml new.kerf.yaml             # path-level diff, values redacted
kerf verify secrets.kerf.yaml                     # integrity check, no plaintext
```

### Run a process with secrets in its environment

```bash
# Decrypts in memory, exports UPPER_SNAKE_CASE env vars, execs. No temp file.
kerf exec secrets.kerf.yaml -- ./my-app --serve
```

### Manage recipients and rotate

```bash
kerf keys add    secrets.kerf.yaml --age age1newteammate...
kerf keys remove secrets.kerf.yaml --age age1formeremployee...
kerf keys list   secrets.kerf.yaml

# Fresh DEK + fresh nonces for every value — the one command that rewrites the whole file
kerf rotate secrets.kerf.yaml --reason "quarterly rotation"
```

## Recipient specifiers

`encrypt` and `keys add`/`keys remove` take recipients as repeatable flags:

| Type | Flag | Example |
|---|---|---|
| AWS KMS | `--kms <arn>` | `--kms arn:aws:kms:us-east-1:111:key/prod-secrets` |
| GCP KMS | `--gcp-kms <id>` | `--gcp-kms projects/p/locations/l/keyRings/r/cryptoKeys/k` |
| Azure Key Vault | `--azure-kv <url>` | `--azure-kv https://vault.vault.azure.net/keys/mykey/version` |
| age | `--age <recipient>` | `--age age1abcdef…` |

`init` records recipients in `.kerf.yaml` instead, using the `<type>:<value>` form (`aws-kms:`, `gcp-kms:`, `azure-kv:`, `age:`; a bare `age1…` is also accepted).

All flags may be repeated for multiple recipients. The recipient flags are intentionally compatible with SOPS so muscle memory carries over.

## Environment variables

To avoid passing recipients on every invocation, kerf reads several env vars. **SOPS-prefixed env vars are honoured as fallback** so existing SOPS users can keep their env setup unchanged.

| Variable | Purpose |
|---|---|
| `KERF_KMS_ARN` / `SOPS_KMS_ARN` | Comma-separated AWS KMS ARNs |
| `KERF_GCP_KMS_IDS` / `SOPS_GCP_KMS_IDS` | Comma-separated GCP KMS resource IDs |
| `KERF_AZURE_KEYVAULT_URLS` / `SOPS_AZURE_KEYVAULT_URLS` | Comma-separated Azure Key Vault key URLs |
| `KERF_AGE_RECIPIENTS` / `SOPS_AGE_RECIPIENTS` | Comma-separated age recipients |
| `KERF_AGE_KEY_FILE` / `SOPS_AGE_KEY_FILE` | Path to age private key file (for decrypt) |
| `KERF_AGE_KEY` / `SOPS_AGE_KEY` | Inline age private key (for decrypt) |

Precedence: explicit CLI flag > `KERF_*` env var > `SOPS_*` env var > `.kerf.yaml` creation rules.

## Planned commands

These appear in [SPEC.md](https://github.com/idestis/kerf/blob/main/SPEC.md) §7 but are not yet in the shipped binary:

- `kerf updatekeys <file>` — reconcile a file's recipients against `.kerf.yaml` (add missing, remove stale) without rotating the DEK.
- `kerf import-sops <file>` — one-time conversion of an existing SOPS file to the kerf format.
